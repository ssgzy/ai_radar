## V1：最小可用版本已完成

### 本版目标

- 跑通 `arXiv` collector
- 跑通 `GitHub` collector
- 跑通本地 Ollama 中文总结
- 输出 `processed JSON`
- 输出 Obsidian 条目笔记和日报
- 保证运行中有明确进度和等待提示

### 本版新增内容

- 完整的项目目录骨架
- 配置层：`sources.yaml`、`models.yaml`、`prompts.yaml`、`obsidian.yaml`
- 数据模型层：`RawItem`、`ProcessedItem`、`SourceRunRecord`
- 采集层：`ArxivCollector`、`GitHubCollector`
- 处理层：`LocalOllamaSummarizer`
- 导出层：`JsonExporter`、`ObsidianExporter`
- 运行入口：`run_once.py`、`run_manual.py`、`run_ai_radar.py`
- 状态落盘：`last_run.json`、`seen_ids.json`、`seen_urls.json`、`seen_titles.json`、`source_checkpoints.json`
- 调试输出：prompt / response 文件
- 基础测试：`test_summarizer.py`、`test_pipeline.py`

### 与上一版差异

- V0 只有核查和文档初始化；V1 已经具备真实可运行的业务闭环
- 新增了模块化模型策略，而不是把模型写死在单脚本中
- 新增了 Obsidian 导出和日报，而不只是 JSON
- 新增了等待心跳提示，长时间模型调用时不会完全安静

### 已解决问题

- 解决了旧脚本结构过散、难以扩展的问题
- 解决了运行结果只落原始 JSON、不落 Obsidian 的问题
- 解决了模型等待时终端几乎无反馈的问题
- 解决了 GitHub 无描述时总结质量明显偏弱的问题

### 仍存在的问题

- 还没有正式的 scoring / tags / dedupe
- 还没有 RSS / Hacker News / News
- 还没有真正的 `launchd` / `cron` 调度脚本
- `qwen3.5:9b` 在当前机器上延迟偏高，不适合作为默认模型

### 下一版计划

- 做 V2：评分、标签、结果排序、失败条目记录、Markdown 结构继续优化

## V2：结构化增强已完成

### 本版目标

- 增加正式的 `scoring / tags`
- 改进 exporter 的 Markdown 结构
- 增强运行记录和日志可读性
- 保留并继续使用 prompt / response 调试输出

### 本版新增内容

- `config/scoring.yaml` 正式启用评分和标签规则
- `src/agents/scorer.py`
- `src/agents/tagger.py`
- `ProcessedItem` 新增评分、标签、推荐级别字段
- `SourceRunRecord` 新增耗时和阶段统计
- `JsonExporter` 新增 `scored JSON`
- `ObsidianExporter` 新增分数排序、V2 结论、标签、分数拆解
- `logs/scorer.log`
- `logs/tagger.log`

### 与 V1 的差异

- V1 只做“总结”；V2 进一步做“打分、打标签、排序”
- V1 的日报只是汇总；V2 的日报会显示优先级最高内容和来源耗时
- V1 的运行记录只有总结果；V2 会记录每个来源的开始时间、结束时间、耗时和阶段统计
- V1 只有 `merged JSON`；V2 新增 `scored JSON`

### 已解决问题

- 解决了 V1 无法直接区分优先级的问题
- 解决了日报不便于快速筛选重点的问题
- 解决了运行记录缺少耗时与阶段信息的问题

### 仍存在的问题

- 当前评分仍是启发式版本，不是模型级排序器
- 标签规则仍以关键词为主，复杂主题聚合能力有限
- 还没有 merge / dedupe 的正式实现
- 还没有接入 `RSS`、`Hacker News`、`News`
